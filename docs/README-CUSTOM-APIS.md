# Custom SCAPI Clients

The storefront communicates with Salesforce B2C Commerce through **SCAPI (Salesforce Commerce API)** clients. The SDK ships with 16 built-in clients (Shopper Products, Shopper Baskets, Shopper Search, etc.), but merchants often deploy their own custom APIs on B2C Commerce — for example, a Loyalty API or a Store Inventory API.

The `sfnext scapi` CLI generates type-safe TypeScript clients for these custom APIs and integrates them with the existing middleware stack (auth, correlation IDs, maintenance detection), so they behave identically to built-in clients.

## Quick Start

### Add a custom API from a local schema

```bash
sfnext scapi add --schema ./my-schemas/loyalty-api.yaml --name loyalty --base-path /custom/loyalty/v1
```

This copies the schema into the project, generates TypeScript types and an operation map, and registers a new `loyalty` client.

### Pull a custom API from B2C Commerce

```bash
sfnext scapi add custom loyalty v1
```

Positional arguments are `<apiFamily> <apiName> <apiVersion>`. The CLI fetches the schema from the SCAPI Schemas API.

## CLI Commands

### `sfnext scapi add`

Add a custom SCAPI client.

**Two modes:**

| Mode | Usage | When to use |
|------|-------|-------------|
| **Pull** | `sfnext scapi add <apiFamily> <apiName> <apiVersion>` | Fetch the schema from B2C Commerce |
| **Local** | `sfnext scapi add --schema <path> --name <key>` | Use a local OpenAPI schema file |

**Flags:**

| Flag | Description |
|------|-------------|
| `--schema <path>` | Path to a local OpenAPI schema file (YAML or JSON) |
| `--name <key>` | Client key name (e.g., `loyalty`, `storeInventory`). Auto-derived from `apiName` in pull mode. |
| `--base-path <path>` | SCAPI base path prefix. Auto-derived from the schema's `servers[].url` if not provided. |
| `--supports-locale` / `--no-supports-locale` | Whether the API accepts a `locale` query parameter (default: `false`). |

**Pull mode environment variables:**

| Variable | Description |
|----------|-------------|
| `SFCC_SHORTCODE` | SCAPI short code |
| `SFCC_TENANT_ID` | Tenant ID |
| `SFCC_OAUTH_CLIENT_ID` | OAuth client ID |
| `SFCC_OAUTH_CLIENT_SECRET` | OAuth client secret |

### `sfnext scapi list`

List registered custom clients:

```bash
sfnext scapi list
```

```
Registered SCAPI clients (1):

  loyalty
    Schema:  schemas/loyalty-v1.yaml
    Base:    /custom/loyalty/v1
    Locale:  no
```

### `sfnext scapi remove`

Remove a registered client and clean up generated files:

```bash
sfnext scapi remove loyalty
```

## How It Works

### Generated File Structure

Running `sfnext scapi add` creates and updates several files:

```
src/scapi/
├── custom-clients.ts                  # Declarative client registry (generated)
├── generated/
│   ├── loyalty-v1.ts                  # TypeScript types from OpenAPI schema
│   └── loyalty-v1.operations.ts       # Operation map (method → HTTP verb + path)
└── schemas/
    ├── loyalty-v1.yaml                # Source OpenAPI schema
    └── loyalty-v1.meta.json           # Metadata sidecar (clientKey, basePath, etc.)
```

> **Do not edit generated files manually.** The `generated/` directory and `custom-clients.ts` are regenerated by the CLI. The `.prettierignore` is configured to skip these files.

### The Client Registry (`src/scapi/custom-clients.ts`)

This file is the bridge between the CLI and the runtime. It declares which custom clients to create and how to configure them:

```typescript
export type AppClients = MergeClients<Clients, {
    loyalty: ProxyClient<Client<P0>, typeof ops0>;
}>;

export const customClients = [
    { key: 'loyalty', basePath: '/custom/loyalty/v1', ops: ops0, locale: false, orgPrefix: true },
] as const;
```

Each entry specifies:

| Field | Description |
|-------|-------------|
| `key` | Property name on the clients object (e.g., `clients.loyalty`) |
| `basePath` | SCAPI base path prefix for all operations |
| `ops` | Generated operation map linking method names to HTTP verbs and paths |
| `locale` | Whether to include `locale` in global query parameters |
| `orgPrefix` | When `true`, injects `/organizations/{organizationId}` into the base URL (custom APIs don't include this in their schema paths) |

The file also exports `AppClients` — a merged type that combines the built-in SDK clients with custom clients. This type is used throughout the application for type-safe API access.

### Runtime Integration (`src/lib/api-clients.ts`)

The `createApiClients()` function reads the client registry and creates custom clients alongside the built-in SDK clients. All clients share the same middleware stack:

- **Auth middleware** — injects Bearer tokens and `sfdc_dwsid`
- **Correlation middleware** — attaches `correlation-id` headers
- **Identifying headers** — sets `x-mobify: true`
- **Maintenance middleware** — detects `sfdc_maintenance` response headers (server-side only)

Custom clients are created using the same `createClient()` / `createOpenApiFetchClient()` primitives as built-in clients, so they get identical behavior.

## Using Custom Clients in Your Code

Custom clients are available on the object returned by `createApiClients()`, with full type safety:

```typescript
import { createApiClients } from '@/lib/api-clients';

export async function loader({ context }: LoaderFunctionArgs) {
    const clients = createApiClients(context);

    // Built-in client (always available)
    const { data: product } = await clients.shopperProducts.getProduct({
        params: { path: { id: 'my-product-id' } },
    });

    // Custom client (available after `sfnext scapi add`)
    const { data: rewards } = await clients.loyalty.getLoyaltyRewards();

    return { product, rewards };
}
```

The `AppClients` type exported from `@/scapi/custom-clients` ensures that `clients.loyalty` is fully typed — autocompletion shows all available operations, and parameters and response types are inferred from the OpenAPI schema.
